\section{Project Overview and Scope}

\subsection{Game Description}

\textit{Orange} is a solo-developed cozy isometric pixel-art puzzle adventure.
The player controls Orange, a chibi isometric character, navigating forest
environments by tap/click. Puzzles are solved through environmental
interactions. Solved puzzles reveal collectible fragments that assemble into
key items, driving the narrative forward.

\subsection{Prototype Scope}

The P01--P30 prompt series targets a minimal but complete vertical slice:

\begin{itemize}
  \item \textbf{Episode 1}: four zones, four puzzles, four fragments that
        complete the \textit{Secret Map Card} key item.
  \item \textbf{Episode 2 Bridge}: one zone, one bridge puzzle that consumes
        the Secret Map Card and triggers a prototype end card.
\end{itemize}

The prototype proves the full reward loop: puzzle solve $\to$ fragment reveal
$\to$ manual collection $\to$ mini-frame fill $\to$ key item completion $\to$
episode transition $\to$ key item use $\to$ end card.

\subsection{Architecture Constraints}

\begin{itemize}[nosep]
  \item Engine: Godot 4.3, GDScript only, no C\#.
  \item Six autoloads in fixed order:
        \texttt{Constants}, \texttt{EventBus}, \texttt{GameState},
        \texttt{SaveSystem}, \texttt{EpisodeController}, \texttt{KeyItemInventory}.
  \item Ten typed global signals via \texttt{EventBus}.
  \item Hybrid data: JSON for narrative text; Godot Resources for gameplay data.
  \item Minimal UI: one action button, one hint level, bottom dialogue strip,
        mini picture frame.
  \item \texttt{SaveSystem} is the sole file writer for \texttt{user://save.json}.
\end{itemize}

  \item All scene files (\texttt{.tscn}) for zones, puzzles, UI, transitions, and fragments.
  \item All puzzle scripts (PuzzleBase + five concrete puzzles).
  \item All UI scripts (HUD, ActionButton, DialogueBox, FloatingLabel, HintButton, MiniFrame, PrototypeEndCard).
  \item All character, camera, interaction, fragment, and key-item scripts.

  \item \texttt{project.godot} (autoload order, renderer, display settings, export).

\subsection{Audit Techniques}

\begin{enumerate}
  \item \textbf{Signal chain tracing}: every \texttt{EventBus} signal was traced
        from emit point through all connected handlers.
  \item \textbf{Godot 4 ready-order analysis}: the bottom-up \texttt{\_ready()}
        call order (children before parents) was applied to identify timing hazards.
  \item \textbf{Scene tree vs. @onready path matching}: each \texttt{@onready} node
        path was matched against the declared scene tree to detect null-dereference risks.
  \item \textbf{Contract verification}: every architecture contract in
        \texttt{prompts.md} §5 was checked against live code.
  \item \textbf{End-to-end flow trace}: the full critical path from Episode~1
        start to end-card display was traced step by step.
\end{enumerate}

\section{Phase A Audit Findings and Resolutions}

\subsection{Issue C-1 (Critical) — PrototypeEndCard Null Dereference}

\subsubsection*{Finding}

\texttt{ui/prototype\_end\_card.gd} declared:

\begin{lstlisting}
@onready var _yes_btn: Button = $ConfirmPanel/VBox/YesButton
@onready var _no_btn:  Button = $ConfirmPanel/VBox/NoButton
\end{lstlisting}

The actual \texttt{.tscn} scene tree places both buttons under an intermediate
\texttt{HBoxContainer} named \texttt{Buttons}:

\begin{lstlisting}
ConfirmPanel
  VBox
    Buttons (HBoxContainer)
      YesButton
      NoButton
\end{lstlisting}

The path mismatch caused Godot to assign \texttt{null} to both \texttt{@onready}
variables. The subsequent \texttt{\_yes\_btn.pressed.connect(...)} in
\texttt{\_ready()} raised a null dereference, halting execution before
\texttt{EventBus.key\_item\_used.connect(\_on\_key\_item\_used)} was reached.
The end card never appeared. The prototype had no observable ending.

\subsubsection*{Impact}

\textbf{Severity}: \sevCrit\quad\textbf{Scope}: P28 end card, P30 prototype end-to-end pass.

\subsubsection*{Fix Applied — FX-001}

\begin{lstlisting}
@onready var _yes_btn: Button = $ConfirmPanel/VBox/Buttons/YesButton
@onready var _no_btn:  Button = $ConfirmPanel/VBox/Buttons/NoButton
\end{lstlisting}

\subsubsection*{Verification}

Both paths confirmed correct in the current repository. \texttt{\_ready()}
completes fully. \texttt{EventBus.key\_item\_used} connection is active.
Status: \sevFixed

\subsection{Issue H-1 (High) — Stale Episode Data in HintButton and InteractionManager}

\subsubsection*{Finding}

Both \texttt{ui/hint\_button.gd} and \texttt{interaction/interaction\_manager.gd}
previously loaded a single episode JSON file in \texttt{\_ready()} by reading
\texttt{GameState.current\_episode}.

In Godot 4, children's \texttt{\_ready()} fires \textit{before} the parent's
\texttt{\_ready()}. \texttt{Zone.\_ready()} updates \texttt{GameState.current\_episode};
HintButton and InteractionManager are both children of the zone node.
Therefore, when loading the Episode~2 bridge zone, both scripts read the stale
\texttt{ep1} episode value and loaded only Episode~1 hint and label data.
The bridge hint button silently showed no hint text. Bridge floating labels
fell back to the scene's \texttt{display\_label} strings.

\subsubsection*{Impact}

\textbf{Severity}: \sevHigh\quad\textbf{Scope}: P13 hint button (bridge zone), P12 floating labels (bridge zone), P27 bridge zone UX.

\subsubsection*{Fix Applied — FX-002}

Both scripts were updated to load \textit{all} known episode JSON paths into
an array and merge the resulting dictionaries, mirroring the pattern already
used by \texttt{DialogueBox}:

\begin{lstlisting}
const _HINT_PATHS: Array[String] = [
    "res://data/hints/episode_1.json",
    "res://data/hints/episode_2_bridge.json",
]
\end{lstlisting}

\begin{lstlisting}
const _LABEL_PATHS: Array[String] = [
    "res://data/floating_labels/episode_1.json",
    "res://data/floating_labels/episode_2_bridge.json",
]
\end{lstlisting}

Godot 4 \texttt{\_ready()}-order sensitivity is permanently eliminated for
both systems.

\subsubsection*{Verification}

Confirmed in current repository. Both scripts use \texttt{\_load\_hint\_data()}
and \texttt{\_load\_label\_data()} that iterate the path arrays and merge.
Status: \sevFixed

\subsection{Issue M-1 (Medium) — EpisodeController.set\_episode() Dead Code}

\subsubsection*{Finding}

\texttt{EpisodeController.set\_episode()} existed but had no callers.
\texttt{current\_episode\_id} in \texttt{EpisodeController} was never updated
after the initial load. When the bridge zone loaded, the MiniFrame was mounted
and \texttt{\_bind\_mini\_frame\_for\_episode()} was called with the stale
\texttt{ep1} ID, binding the bridge zone's MiniFrame to the already-completed
Secret Map Card rather than the Mixed Compass.

\subsubsection*{Impact}

\textbf{Severity}: \sevMed\quad\textbf{Scope}: P20 episode tracking, P17 MiniFrame in bridge zone, P27 bridge zone.

\subsubsection*{Fix Applied — FX-003}

\texttt{\_on\_node\_added} in \texttt{EpisodeController} was extended to
detect incoming \texttt{Zone} nodes and call \texttt{set\_episode()} from
the zone's \texttt{zone\_id} prefix:

\begin{lstlisting}
func _on_node_added(node: Node) -> void:
    if node is Zone:
        var zid := str((node as Zone).zone_id)
        if zid.begins_with("ep2bridge"):
            set_episode(&"ep2bridge")
        elif zid.begins_with("ep1"):
            set_episode(&"ep1")
    if not (node is Hud):
        return
    ...
\end{lstlisting}

\subsubsection*{Verification}

Confirmed in current repository. Zone detection fires before HUD detection in
the same \texttt{node\_added} handler. \texttt{set\_episode()} sets
\texttt{current\_episode\_id} and calls \texttt{\_bind\_mini\_frame\_for\_episode()},
which rebinds an existing MiniFrame if already mounted.
Status: \sevFixed

\textbf{Additional note}: \texttt{zone.gd}~\texttt{\_ready()} was also updated
to set \texttt{GameState.current\_episode} directly from the zone's
\texttt{zone\_id} prefix. This belt-and-suspenders approach ensures
\texttt{GameState.current\_episode} is always correct for any system reading it
after \texttt{\_ready()} completes, independent of EpisodeController timing.

\subsection{Issue M-2 (Medium) — PrototypeEndCard Under Node2D Root}

\subsubsection*{Finding}

\texttt{PrototypeEndCard} (a \texttt{Control} node) was parented directly
under the \texttt{ZoneSunlitClearing} \texttt{Node2D} root. Control nodes
that require reliable full-screen anchoring must be under a
\texttt{CanvasLayer} or a \texttt{Control} parent that covers the viewport.
Parenting under a world-space \texttt{Node2D} leaves anchoring and position
relative to 2D world coordinates, not the viewport, causing unreliable layout
in web and mobile scaling contexts.

\subsubsection*{Impact}

\textbf{Severity}: \sevMed\quad\textbf{Scope}: P28 end card layout in web/mobile export.

\subsubsection*{Fix Applied — FX-004}

An \texttt{EndCardLayer} (\texttt{CanvasLayer}, \texttt{layer = 64}) was
added to the bridge zone scene. \texttt{PrototypeEndCard} was re-parented
under it. Layer 64 places it above all gameplay (layer~0) and HUD (layer~10)
but below the \texttt{SoftTransition} overlay (layer~128).

\begin{lstlisting}
[node name="EndCardLayer" type="CanvasLayer" parent="."]
layer = 64

[node name="PrototypeEndCard" parent="EndCardLayer"
      instance=ExtResource("10_end")]
\end{lstlisting}

\subsubsection*{Verification}

Confirmed in current \texttt{zone\_sunlit\_clearing.tscn}.
Status: \sevFixed

\section{Remaining Low-Priority Observations}

The following items are informational and do not block the prototype. They
are recorded here as candidates for future improvement.

\begin{enumerate}

  \item \textbf{MapCompletionTrigger primary MiniFrame path} — The primary
  lookup path \texttt{hud.get\_node\_or\_null("CanvasLayer/MiniFrame")}
  does not match the actual HUD slot structure. The fallback
  \texttt{find\_child("MiniFrame")} succeeds at runtime. The primary path
  should be updated to match \texttt{EpisodeController}'s actual mount
  structure in a future maintenance pass.

  \item \textbf{PrototypeEndCard direct scene change} — \texttt{\_on\_confirm\_yes}
  calls \texttt{get\_tree().change\_scene\_to\_file(zone1\_path)} directly,
  bypassing \texttt{EpisodeController}. This is acceptable for the prototype
  restart path but should be routed through \texttt{EventBus.zone\_change\_requested}
  if the restart logic grows.

  \item \textbf{SunlitMapActivationPuzzle hardcoded flag key} — The completion
  flag is stored as the raw string \texttt{"ep2bridge\_completed"} rather than
  a \texttt{Constants} entry. No functional impact for the prototype; add a
  constant if additional modules need to query this flag.

  \item \textbf{FragmentPickup wiring per zone} — \texttt{zone.gd~\_setup\_hud()}
  does not auto-connect \texttt{action\_dispatched} to \texttt{FragmentPickup}
  nodes. Zone scene designers must connect this manually or via the scene
  editor. Document this in a design note to prevent future confusion.

  \item \textbf{Web export preset policy} — \texttt{export\_presets.cfg} is
  present in the working tree but untracked. The C06 clarification recommends
  committing it if it contains no secrets. Resolve this policy explicitly in
  the P29 commit.

  \item \textbf{Hint JSON values are plain strings} — The hint JSON files use
  plain string values rather than dictionaries with \texttt{orange}/\texttt{english}
  keys. This is consistent with the current \texttt{HintButton} implementation
  and is not a bug, but if hints ever need fictional-language variants the
  schema will need to be extended.

\end{enumerate}
